# CASE - 知识库处理

本目录演示 RAG 系统中**知识库全生命周期管理**的四个核心环节：问题生成与检索优化、对话知识沉淀、健康度检查、版本管理与性能比较。场景均以「上海迪士尼乐园」知识为例，覆盖知识入库前的增强、运营中的质量治理、以及版本迭代时的效果评估。

## 脚本说明

本案例由 4 个独立脚本组成，各自聚焦知识库处理的一个维度：

| 脚本 | 主题 | 核心技术 |
|------|------|----------|
| 1-知识库问题生成与检索优化-BM25.py | 问题增强检索 | LLM 生成问题 + BM25 双索引 |
| 2-对话知识沉淀.py | 对话转知识 | LLM 提取 + 合并去重 |
| 3-知识库健康度检查.py | 质量治理 | LLM 三维体检 + 健康评分 |
| 4-知识库版本管理与性能比较.py | 版本迭代 | FAISS 向量检索 + 版本 diff + 回归测试 |

---

### 1-知识库问题生成与检索优化-BM25.py

**作用：** 为每个知识切片**自动生成多样化问题**，构建「原文索引 + 问题索引」双 BM25 检索通道，对比两种检索方式的准确率。

**核心类：`KnowledgeBaseOptimizer`**

| 方法 | 说明 |
|------|------|
| `generate_questions_for_chunk` | 为单个切片生成 5 个多样化问题（直接问/间接问/对比问等） |
| `generate_diverse_questions` | 生成 8 个更丰富的问题（含提问角度、能否回答、参考答案） |
| `build_knowledge_index` | 构建原文 BM25 索引与问题 BM25 索引 |
| `search_similar_chunks` | 按 `search_type`（`content` / `question`）检索 |
| `evaluate_retrieval_methods` | 对比原文检索 vs 问题检索的 Top-1 准确率 |

**检索优化原理：**

用户提问往往更接近「问题表述」，而非知识库中的「陈述句」。为每个知识块预生成问题并单独建索引，可缩小 query 与文档之间的语义鸿沟。

```
知识切片原文  ──→ BM25 原文索引
     │
     └── LLM 生成 N 个问题  ──→ BM25 问题索引（内容+问题拼接分词）
```

**文本预处理：** `jieba` 分词 + 停用词过滤 + 标点去除

**内置示例：** 8 条迪士尼知识 + 3 组挑战性测试查询（间接问法）

---

### 2-对话知识沉淀.py

**作用：** 从客服/用户**多轮对话**中自动提取结构化知识，批量处理后经 LLM 合并相似条目，形成可入库的知识候选。

**核心类：`ConversationKnowledgeExtractor`**

| 方法 | 说明 |
|------|------|
| `extract_knowledge_from_conversation` | 单次对话提取：事实/需求/问题/流程/注意 五类知识 |
| `batch_extract_knowledge` | 批量提取并统计知识出现频率 |
| `merge_similar_knowledge` | 过滤「需求」「问题」类临时知识，按类型分组后 LLM 合并 |
| `merge_knowledge_with_llm` | 同类型多条知识智能合并为一条 |

**提取输出结构（JSON）：**

```json
{
  "extracted_knowledge": [
    {
      "knowledge_type": "事实",
      "content": "知识内容",
      "confidence": 0.9,
      "source": "AI",
      "keywords": ["门票", "价格"],
      "category": "价格信息"
    }
  ],
  "conversation_summary": "对话摘要",
  "user_intent": "用户意图"
}
```

**沉淀策略：**

- 「需求」「问题」类为个性化、临时信息，合并前**自动过滤**
- 「事实」「流程」「注意」类沉淀为长期知识
- 合并时保留最高置信度，整合关键词与来源

**适用场景：** 客服机器人对话日志 → 知识库增量更新

---

### 3-知识库健康度检查.py

**作用：** 对知识库进行**三维健康检查**，生成覆盖率、新鲜度、一致性评分及改进建议。

**核心类：`KnowledgeBaseHealthChecker`**

| 检查维度 | 方法 | 检查内容 |
|----------|------|----------|
| 完整性 | `check_missing_knowledge` | 测试查询能否在知识库中找到答案，识别知识空白 |
| 时效性 | `check_outdated_knowledge` | 价格、政策、活动时间等是否过期 |
| 一致性 | `check_conflicting_knowledge` | 同一主题是否存在矛盾表述（如票价不一致） |

**整体健康度计算：**

```
overall_score = coverage × 0.4 + freshness × 0.3 + consistency × 0.3
```

| 分数区间 | 健康等级 |
|----------|----------|
| ≥ 0.8 | 优秀 |
| ≥ 0.6 | 良好 |
| ≥ 0.4 | 一般 |
| < 0.4 | 需要改进 |

**内置示例：** 故意构造了票价冲突（kb_002 vs kb_003）、过期更新时间、缺少活动/停车费知识等场景，便于观察检查结果。

---

### 4-知识库版本管理与性能比较.py

**作用：** 管理知识库**多版本**，支持版本 diff、向量检索性能评估、版本间对比及回归测试。

**核心类：`KnowledgeBaseVersionManager`**

| 功能 | 方法 | 说明 |
|------|------|------|
| 创建版本 | `create_version` | 构建 FAISS 向量索引，记录统计信息 |
| 版本对比 | `compare_versions` | 检测新增/删除/修改/未变的知识切片 |
| 性能评估 | `evaluate_version_performance` | 对测试查询集评估检索准确率与响应时间 |
| 版本性能对比 | `compare_version_performance` | 两版本准确率、耗时对比 + 推荐建议 |
| 回归测试 | `generate_regression_test` | 用固定测试集验证新版本是否退化 |

**向量检索配置：**

| 项目 | 值 |
|------|-----|
| Embedding 模型 | text-embedding-v4 |
| 向量维度 | 1024 |
| 索引类型 | FAISS IndexFlatL2 + IndexIDMap |

**准确率判定：** 检索到的 chunk 内容中**包含** `expected_answer` 关键词即视为正确（简化评估）。

**内置示例：**

- v1.0：3 条基础迪士尼知识
- v2.0：5 条增强知识（更详细票价、交通、游乐项目等）
- 5 组测试查询用于性能对比与回归测试

## 技术选型

### 大语言模型：qwen-turbo-latest

- 脚本 1：OpenAI 兼容接口（`client.chat.completions.create`）
- 脚本 2、3、4：`dashscope.Generation.call`
- 用于问题生成、知识提取、健康检查、知识合并

> 若 `qwen-turbo-latest` 返回 403，可改为 `qwen-plus`（与项目其他 CASE 一致）。

### BM25 检索（脚本 1）

- `rank_bm25.BM25Okapi` + `jieba` 中文分词
- 双索引：原文索引 + 问题索引

### 向量检索（脚本 4）

- `text-embedding-v4` 通过 OpenAI 兼容接口获取 Embedding
- `faiss.IndexFlatL2` 构建版本内向量索引

### 数据处理

- `pandas`：版本统计与对比（脚本 4 间接使用）
- `numpy`：向量数组运算

## 环境准备

### 1. 安装依赖

建议在项目根目录虚拟环境中安装：

```powershell
# 激活虚拟环境（项目根目录 d:\ai_script）
.\.venv\Scripts\Activate.ps1

pip install -r requirements.txt
```

| 包名 | 版本 | 用途 |
|------|------|------|
| dashscope | 1.25.5 | LLM 调用（脚本 2、3、4） |
| openai | 2.14.0 | 兼容接口调用 LLM / Embedding（脚本 1、4） |
| rank_bm25 | 0.2.2 | BM25 检索（脚本 1） |
| jieba | 0.42.1 | 中文分词（脚本 1） |
| faiss-cpu | 1.7.4 | 向量索引（脚本 4） |
| pandas | 2.3.3 | 数据处理 |
| numpy | 见下方说明 | 向量运算 |

- `requirements.txt` 中 `numpy==2.4.0` 与 `faiss_cpu==1.7.4` 在 Windows 上不兼容，实际环境需使用 **`numpy<2`（如 1.26.4）**。

### 2. 配置 API Key

```powershell

```

脚本 1~4 均依赖此环境变量。

## 运行方式

脚本内置迪士尼示例数据，无需外部文件，可在任意目录运行；建议在案例目录下执行以便管理：

```powershell
cd d:\ai_script\04_rag_tuning\CASE-知识库处理

# 脚本 1：问题生成 + BM25 检索对比（调用 LLM 较多，耗时较长）
python 1-知识库问题生成与检索优化-BM25.py

# 脚本 2：对话知识提取与沉淀
python 2-对话知识沉淀.py

# 脚本 3：知识库健康度检查
python 3-知识库健康度检查.py

# 脚本 4：版本管理与性能比较（含 Embedding API 调用）
python 4-知识库版本管理与性能比较.py
```

### 各脚本运行说明

| 脚本 | 主要输出 | 预计 API 调用 |
|------|----------|---------------|
| 1 | 生成的问题列表、原文 vs 问题检索准确率对比 | 较多（每个 chunk 生成问题） |
| 2 | 提取的知识点、合并后的沉淀结果 | 中等 |
| 3 | 健康度评分、缺失/过期/冲突详情、改进建议 | 3 次 LLM 调用 |
| 4 | 版本 diff、准确率、响应时间、回归测试通过率 | 含 Embedding 调用 |

## 常见问题

### 1. `DASHSCOPE_API_KEY` 未设置

PowerShell 中执行 `$env:DASHSCOPE_API_KEY = "sk-xxx"` 后重试。

### 2. JSON 解析失败

LLM 返回可能带 Markdown 代码块。脚本 1、2 已实现 `preprocess_json_response` 清理；脚本 3 在解析前也会剥离 ` ``` ` 标记。仍失败时可改用 `qwen-plus` 或降低 `temperature`。

### 3. 脚本 1 运行很慢

`evaluate_retrieval_methods` 会为每个知识切片调用 LLM 生成问题（8 个 chunk × 1 次 API）。测试时可减少 `knowledge_base` 条目或注释掉示例 3。

### 4. faiss 导入报错

执行 `pip install "numpy<2"` 解决与 faiss-cpu 1.7.4 的兼容问题。

### 5. 脚本 4 Embedding 失败

确认账号支持 `text-embedding-v4` 模型；若不可用，可将 `TEXT_EMBEDDING_MODEL` 改为 `text-embedding-v1` 并同步调整 `TEXT_EMBEDDING_DIM`。

### 6. 健康检查结果是 LLM 判断

脚本 3 的覆盖率、新鲜度、一致性评分由大模型主观评估，适合演示流程；生产环境建议结合规则引擎与人工抽检。

### 7. 版本性能评估的准确率较简化

脚本 4 以「检索结果是否包含期望关键词」判定正确，非语义级评估。实际项目可引入更严格的评测集与人工标注。
